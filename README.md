# LLM Research Assistant

A hybrid-search RAG system that answers ML/LLM research questions using cited, independently verified evidence from arXiv papers. Built from scratch in plain Python, no LangChain, no framework abstractions, to understand and control every stage of the pipeline: ingestion, chunking, dense + sparse retrieval, reranking, generation, citation, and verification.

The core focus of this project isn't "can an LLM answer questions about papers", it's **can you trust the citations it gives you**. Every claim in every answer is checked, after generation, by a second model that never saw the generation step, against the exact source text it claims to be citing.

![Citation verification catching a subtle oversimplification](docs/screenshots/citation-verification.png)

In the example above, the system correctly distinguished between two claims in the same answer: one accurately summarized the source (✅ supported), the other oversimplified a conditional, iterative architecture into a clean linear pipeline (⚠️ partially supported), a distinction a casual reader would likely miss without checking the source themselves.

## How it works

```
arXiv papers
   │
   ▼
Ingestion (arXiv API + PDF extraction)
   │
   ▼
Cleaning (reference/bibliography stripping)
   │
   ▼
Chunking (sentence-aware, token-based, overlapping)
   │
   ├──────────────┐
   ▼              ▼
Dense embedding   BM25 sparse index
(BGE-M3 → Chroma)  (rank_bm25)
   │              │
   └──────┬───────┘
          ▼
  Reciprocal Rank Fusion (hybrid retrieval)
          ▼
  Cross-encoder reranking (bge-reranker-v2-m3)
          ▼
  Context construction (token-budgeted, citation-mapped)
          ▼
  Answer generation (Groq · Llama-3.3-70B)
          ▼
  Citation formatting (per-document [N] references)
          ▼
  Citation verification (Claude Haiku 4.5, independent judge)
          ▼
  Answer + references + per-sentence verdicts
```

Served via a FastAPI backend (models loaded once at startup, not per request) and a Streamlit chat frontend.

## Why these design choices

**Hybrid retrieval, not just dense search.** Dense embeddings (BGE-M3) capture semantic similarity; BM25 captures exact lexical/keyword matches. They're fused via Reciprocal Rank Fusion (`1/(k+rank)`, k=60) rather than one replacing the other, since they fail in different, complementary ways.

**Reranking measurably helps, this isn't just assumed.** A cross-encoder reads the query and each candidate chunk *together* (rather than independently, like embeddings do), which catches relevance signals dense/sparse retrieval alone miss. On a 6-query hand-labeled evaluation set across the full corpus:

| Stage | MRR |
|---|---|
| Hybrid search only | 0.6905 |
| + Cross-encoder reranking | 0.8333 |

**Citations are collapsed at the document level for display, but tracked at the chunk level internally.** Multiple chunks from the same paper share one `[N]` marker (like a reference list, `[1]` can appear many times in the body but only once in the bibliography), so answers don't look repetitive. But every contributing chunk stays individually tracked, which is what makes chunk-level citation verification possible.

**Verification uses a genuinely independent model.** Answers are generated by Groq's Llama-3.3-70B; verification is done by Claude Haiku 4.5, a different lab, different architecture, different training. Using the same model to grade its own answer risks it simply agreeing with itself. Cross-lab verification is a stronger (though not perfect, see Known Limitations) check.

**A token budget caps context size regardless of chunk size.** Some chunks in the corpus are much larger than others (long tables, occasional unstripped reference lists, see below). Context construction caps the total request at ~6,000 tokens, keeping/dropping chunks in reranked order, so a single request never balloons unpredictably.

## The corpus

241 papers pulled from the arXiv API across three topic queries: `"large language models"`, `"retrieval augmented generation"`, and `"agentic AI"`, roughly 9,400 chunks after cleaning and chunking. Corpus data isn't committed to this repo (see Setup below for how to regenerate it), only the pipeline code is.

## Known limitations

Documenting these deliberately, rather than pretending they don't exist:

- **~4.6% of papers (11/241) retain an unstripped references section**, due to PDF-to-markdown extraction inconsistencies. 6 of those are fixable with a regex extension (numbered/compound headings like `## 9. References` aren't currently matched); the other 5 never had a references heading captured by extraction at all, which no cleaning-stage fix can address. Citation verification mitigates the downstream impact, a bibliography chunk cited as "evidence" for a real claim would almost always be flagged as unsupported.
- **Retrieval has no relevance threshold.** Hybrid search and reranking always return their top-K candidates regardless of how weakly relevant they are. For genuinely out-of-corpus questions, the only safeguard against hallucination is the generation model honoring its grounding instructions, which held up in testing, but is a prompt-level guarantee, not a retrieval-level one.
- **Strict grounding trades recall for precision.** A query paraphrased far enough from a paper's own terminology can cause the model to decline to answer even when the correct chunk was successfully retrieved and present in its context, observed directly during testing. This is a deliberate tradeoff: the system is built to prefer "I don't know" over a confident but disconnected answer.

## Setup

```bash
git clone <this repo>
cd LLM_Research_Assistant
python -m venv venv
source venv/bin/activate   # Windows: venv\Scripts\activate
pip install -r requirements.txt
python -c "import nltk; nltk.download('punkt_tab')"
```

Copy `.env.example` to `.env` and fill in your own API keys:

```bash
cp .env.example .env
# then edit .env, needs GROQ_API_KEY and ANTHROPIC_API_KEY
```

The `data/` folder is not included in this repo, regenerate it by running the pipeline end to end, in order:

```bash
python -m src.ingestion.arxiv      # pulls papers from the arXiv API
python -m src.cleaning.clean
python -m src.chunking.chunk
python -m src.embedding.embed      # builds the Chroma vector store
python -m src.retrieval.bm25       # builds the BM25 index
```

Then run the backend and frontend (two separate terminals):

```bash
python -m src.api.main             # FastAPI, http://localhost:8000
streamlit run src/frontend/app.py  # opens in your browser
```

Optionally, check retrieval quality:

```bash
python -m src.evaluation.evaluate
```

## Project structure

```
src/
  schemas.py              Pydantic models shared across every stage
  ingestion/arxiv.py       arXiv API search + PDF extraction
  cleaning/clean.py        Reference/bibliography stripping
  chunking/chunk.py        Sentence-aware, token-based overlapping chunks
  embedding/embed.py       BGE-M3 embeddings → Chroma
  retrieval/
    bm25.py                Sparse (lexical) index
    hybrid.py               Dense + sparse fusion via RRF
    rerank.py               Cross-encoder reranking
  context/context.py       Token-budgeted context + citation mapping
  generation/
    generate.py             Groq Llama-3.3-70B answer generation
    cite.py                 Citation formatting
  verification/verify.py   Claude Haiku 4.5 independent claim verification
  evaluation/evaluate.py   MRR evaluation (hybrid vs. reranked)
  api/main.py               FastAPI backend
  frontend/app.py           Streamlit chat interface
  logging_config.py        Shared logging setup (console + rotating file)
docs/screenshots/          README images
```
